# Proton Drive Remote Backend

`backup` recognizes `protondrive://` remotes through an Ada SDK layer for the Proton Drive integration contract. This backend is provider-gated by default: it parses configuration, validates the required SDK wiring, derives metadata authentication material with the local CryptoLib library, and performs upload/download/list/delete/event calls only when the session descriptor declares explicit Proton Drive operation endpoint templates. A descriptor may opt into the Ada native endpoint surface with `proton_drive_native_api=true`; that mode derives endpoint templates from the official API base and requires stronger node/content key material before it is accepted. Descriptors may declare `proton_drive_descriptor_version=1` and native mode may declare `proton_drive_sdk_generation=ada-compat-v1` and must declare `proton_drive_wire_contract=proton-drive-sdk-compat-v1`; unsupported descriptor, SDK, or wire-contract generations fail closed. Without either explicit endpoints or native opt-in it remains fail-closed.

ProtonDriveApps/sdk PR #23 documents the upstream Proton Drive SDK as MIT licensed and currently providing TypeScript, C#, Kotlin, and Swift implementations. Proton documents the SDK as focused on first-party clients, allowed for personal non-commercial projects, not ready for commercial or production third-party use, and subject to interface and cryptographic-model changes. The SDK README also states that it does not include authentication/login flows, session management, or a user address provider. Backup therefore must not make raw Proton Drive API calls that bypass the SDK encryption and metadata model.

## URL Form

```sh
protondrive://SHARE_ID/backups/
protondrive://SHARE_ID/backups/nightly.zip
```

The first component is the Proton Drive share id. The remaining path is the backup-managed namespace or object name.

## Configuration

```conf
remote=protondrive://SHARE_ID/backups/
proton_drive_app_version=external-drive-backup@0.1.0-alpha
proton_drive_session_file=/secure/path/proton-session.json
proton_drive_user_address=user@example.com
# Optional, defaults to the official Proton Drive host used by the Ada SDK layer:
proton_drive_api_base=https://drive.proton.me
```

`proton_drive_app_version` must follow ProtonDriveApps/sdk PR #23's published `x-pm-appversion` shape: `external-drive-{name}@{semver}-{channel}+{suffix}`, where `+{suffix}` is optional and channel is `stable`, `beta`, or `alpha`; for example `external-drive-backup@0.1.0-alpha` or `external-drive-photo_backup@1.2.3-stable+abc123f`. The session file is a local provider descriptor. The current Ada SDK layer reads `uid`, `access_token`, optional `refresh_token`, `user_address`, `address_id`, `address_key_id`, `address_key_material`, optional `address_key_fingerprint`, `metadata_key_id`, `metadata_hmac_key`, optional `node_key_material`, optional `content_key_material`, and optional `content_hmac_key` from this JSON file. Key fields may instead be stored as `encrypted_address_key_material`, `encrypted_metadata_hmac_key`, `encrypted_node_key_material`, `encrypted_content_key_material`, and `encrypted_content_hmac_key`; backup unlocks those with `proton_drive_key_unlock_passphrase`, `proton_drive_key_unlock_salt`, and `proton_drive_key_unlock_rounds` using CryptoLib BCrypt-PBKDF plus ChaCha20-Poly1305 and fails closed when the unlock tag does not verify. Native API mode requires the node/content fields; provider-template mode keeps them optional for compatibility with external Proton-compatible providers. `proton_drive_user_address` must match the descriptor when `user_address` is present, and the descriptor must include address/key ids so crypto work binds metadata to the correct Proton address key. Backup can call explicit provider-declared refresh/login/session endpoints through `Refresh_Session`, `Start_Login`, `Complete_MFA`, and `Bootstrap_Session`, and `Login_And_Save_Session` runs that configured login/MFA/bootstrap flow and rewrites the local descriptor with the returned tokens while preserving endpoint and key fields. The `backup --proton-drive-login SESSION_FILE USER_ADDRESS USERNAME PASSWORD_PROOF [MFA_CODE] [SESSION_LABEL] [APP_VERSION] [API_BASE]` helper exposes this flow for users. This remains provider-gated because the upstream SDK does not provide a standalone login/session manager or user address provider. Native API mode does not derive first-party auth endpoints; `--proton-drive-login` requires explicit auth-provider endpoint templates or a provider operation base. Direct first-party auth is unsupported and fails closed. The descriptor may also include `proton_drive_operation_base_url` or per-operation templates such as `proton_drive_session_bootstrap_url`: `proton_drive_upload_url`, `proton_drive_download_url`, `proton_drive_delete_url`, `proton_drive_list_url`, `proton_drive_events_url`, `proton_drive_refresh_url`, `proton_drive_create_folder_url`, `proton_drive_conflict_url`, `proton_drive_list_continue_url`, `proton_drive_events_continue_url`, `proton_drive_trash_url`, `proton_drive_revision_url`, `proton_drive_upload_start_url`, `proton_drive_upload_chunk_url`, `proton_drive_upload_finish_url`, and `proton_drive_resume_upload_url`. Templates may use `{share_id}`, `{parent_path}`, `{remote_name}`, `{remote_path}`, `{upload_id}`, `{part_number}`, and `{after}`. A descriptor can advertise opt-in compatibility checks with `proton_drive_live_check=true` or `proton_drive_live_check_url=...`; release tests do not contact Proton unless such external credentials/endpoints are supplied.

The Ada SDK layer uses `CryptoLib.Hashes.SHA256` to derive an address-key fingerprint from `uid`, `address_id`, `address_key_id`, and `address_key_material`. When `address_key_fingerprint` is present, it must match the derived value. It also derives node/content key fingerprints when the descriptor supplies `node_key_material`, `content_key_material`, and `content_hmac_key`; native mode fails closed without them. Metadata is wrapped as a deterministic `Metadata_Envelope` for provider compatibility. `Encrypted_Metadata_Envelope` now emits `backup-proton-metadata-v2` envelopes sealed with CryptoLib ChaCha20-Poly1305 over the canonical metadata and metadata authentication tag, and `Open_Encrypted_Metadata_Envelope` verifies and opens those envelopes. This still uses descriptor-provided or descriptor-unlocked key material. `Supports_First_Party_Key_Unlock` intentionally returns false and `First_Party_Key_Unlock_Status` reports the upstream limitation until Proton publishes a standalone account key provider surface for third-party clients. Upload payloads with content key material are wrapped through `Streaming_Content_Block` as `PROTON-STREAM-BLOCK-V1` envelopes: each block gets a derived nonce, deterministic keystream encryption, hex ciphertext, and an HMAC over the encrypted block metadata and ciphertext. `Open_Streaming_Content_Block` verifies and decrypts those envelopes on download. Older provider envelopes still use deterministic content-block HMAC metadata (`Content_Block_Tag`) and `Tagged_Content_Block` so provider/native endpoints can reject mismatched block ordering or payload corruption before accepting data. Downloads that include either a `PROTON-STREAM-BLOCK-V1` envelope or a `PROTON-BLOCK-TAG:` prefix are verified before the archive is written. Files above `proton_drive_large_upload_threshold` require chunk endpoints; `Plan_Upload` and `Requires_Chunked_Upload` expose that bounded-memory decision and backup fails closed instead of reading a large archive into memory as one payload. It uses `CryptoLib.Macs.HMAC_SHA256` to derive deterministic authentication tags for canonical metadata packets from the provider-supplied `metadata_hmac_key`. These fields are provider input until full Proton-compatible key unlocking is implemented.

## Current Status

The Ada package `Proton_Drive` now provides client configuration, session loading, optional token refresh through a provider endpoint, user-address/key provider resolution, descriptor schema validation, encrypted descriptor key-material unlocking, auth/session capability validation including provider-only `Supports_Native_Auth_Flow`, descriptor login updates through `Login_And_Save_Session`, CryptoLib-backed crypto context loading, canonical metadata authentication, ChaCha20-Poly1305 encrypted metadata envelopes, content-block authentication tags and envelopes, validation, node metadata, event cursor, provider-backed or explicit native upload/download/list/delete/event operation contracts with required response-field validation through `Wire_Response_Valid`, folder/conflict/resume/trash/revision/event-replay capability detection plus callable `Create_Folder`, `Resolve_Conflict`, `Trash_Node`, `Latest_Revision`, and `Resume_Upload` operations, pagination/continuation hooks, live compatibility gate integration, and chunked large-upload contracts through `proton_drive_upload_start_url`, `proton_drive_upload_chunk_url`, and `proton_drive_upload_finish_url` or native API mode. `backup` routes `protondrive://` upload, verification download, restore download, and delete through this SDK layer. Missing key material returns typed `SDK_Crypto_Unavailable` failures. Missing operation endpoints return typed `SDK_Operations_Unavailable` failures.

## Release Guard

`tools/bin/check_all` verifies that the Proton Drive transport, Ada SDK layer, configuration keys, documentation, native API opt-in, descriptor schema validation, encrypted key-material unlock checks, explicit wire contract, live-check hooks, folder/conflict/resume hooks, encrypted metadata envelopes, content-block tags, bounded-memory large-upload behavior, large-upload surface, and fail-closed tests remain present. `tools/bin/check_proton_drive_compatibility` provides the opt-in live gate; set `BACKUP_PROTON_DRIVE_COMPAT_REMOTE`, `BACKUP_PROTON_DRIVE_COMPAT_SESSION_FILE`, and `BACKUP_PROTON_DRIVE_COMPAT_USER_ADDRESS` for a real provider/account, or `BACKUP_PROTON_DRIVE_COMPAT_ALLOW_FIXTURE=1` for the offline fixture in strict CI. Release checks verify the provider-gated operation surface. Real Proton-provider compatibility remains opt-in because callers must supply provider endpoint templates for auth and either provider operation templates or explicitly enable native Drive endpoint derivation plus complete session/key material.

## Provider Protocol Surface

The Ada provider contract is intentionally explicit. PR #23's C# SDK surface names `FilesApiClient`, `FoldersApiClient`, `LinksApiClient`, `TrashApiClient`, block upload preparation/verification DTOs including `BlockUploadPreparationRequest`, revision DTOs, upload/download controllers, and interop error conversion; the Ada native/provider endpoint layer should keep request and response contracts aligned with those surfaces as they stabilize.

The Ada provider contract is intentionally explicit. Auth providers implement login/MFA/bootstrap/refresh endpoints used by `Start_Login`, `Complete_MFA`, `Bootstrap_Session`, `Login_And_Save_Session`, and `Refresh_Session`; Drive providers implement upload/download/delete/list/events plus optional `Create_Folder`, `Resolve_Conflict`, `Trash_Node`, `Latest_Revision`, and `Resume_Upload`. Uploads use `Plan_Upload` for bounded-memory chunking, `Streaming_Content_Block` for encrypted per-block envelopes where content keys are present, and `Tagged_Content_Block` for compatibility fallback authentication. Metadata uses `Encrypted_Metadata_Envelope` so a future Proton-compatible encryption implementation can replace the current deterministic provider envelope without changing backup call sites.
