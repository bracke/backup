# SPARK and GNATprove

`backup` has committed `SPARK_Mode => On` proof surfaces in `Backup.Path_Syntax`, `Backup.Incremental_Syntax`, `Backup.Jobs_Syntax`, `Backup.Jobs_Retention_Syntax`, `Backup.Remote_Syntax`, `Backup.Remote_Sync_Syntax`, `Backup.Catalog_Syntax`, `Backup.Restore_Syntax`, `Backup.Manifest_Syntax`, `Backup.Zip_Syntax`, and `Backup.CLI_Syntax`. `Backup.Path_Syntax` covers pure path, archive-object, managed-object, and object-name safety predicates shared by higher-level packages. `Backup.Incremental_Syntax` covers pure incremental status, decision, entry-kind, and ZIP method naming helpers used by reports and diagnostics. `Backup.Jobs_Syntax` covers pure job configuration syntax decisions for repeated keys, booleans, natural numbers, schedule metadata, and job status text. `Backup.Jobs_Retention_Syntax` covers pure retention keep-target, count-policy deletion, and tiered bucket capacity decisions. `Backup.Remote_Syntax` covers pure remote status, transport, sync-action, HTTP status, transport classification, resume-mode, timeout precheck, and retry-decision helpers. `Backup.Remote_Sync_Syntax` covers pure remote sync action-selection rules for inventory items and missing local archives. `Backup.Restore_Syntax` covers pure restore status text, action text, dry-run report-action mapping, restore path filter matching, and safe symlink target validation. `Backup.Catalog_Syntax` covers pure catalog status, verification, encryption, entry-kind, numeric, boolean, and query-value parse helpers. `Backup.Manifest_Syntax` covers pure manifest result, entry-kind, and compression-method naming used by manifest reports. `Backup.Zip_Syntax` covers pure ZIP version and general-purpose flag admission used by archive verification. `Backup.CLI_Syntax` covers pure decimal digit recognition, overflow-checked byte-count accumulation, catalog/job/remote command selection predicates, restore conflict-policy admission, and positional-path admission decisions used by CLI validation. `Backup.CLI_Surface` is generated from the shared CLI surface model with `SPARK_Mode => On`, keeping executable help ordering and role selection in the same generated surface as the catalog, man page, and completions. Release validation runs GNATprove in level-4 proof mode through `backup_spark.gpr` so future SPARK annotations cannot regress silently while dependency proofs remain owned by their own crates.

```sh
alr exec -- gnatprove -P backup_spark.gpr --level=4
```

When pure value-level helpers are introduced, enable `SPARK_Mode => On` locally and raise this gate only after the checked surface proves cleanly.
